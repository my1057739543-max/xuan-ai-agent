# 11. Version 1.2 路线与边界（游戏助手增强版）

# 你必须使用spring ai这个框架开发如果涉及agent开发部分

## 1. 文档目标

本篇作为 version1.2 的总执行入口，给出：

- 业务目标与版本边界
- 模块拆分与依赖顺序
- 每个阶段的输入输出
- 验收口径与风险控制

本阶段基线来自 project1.2.md，且必须保持以下原则：

1. 必须继续使用 Spring AI 生态能力实现 Agent/RAG 主链路。
2. 以业务正确性优先，重点解决跨游戏误答与低可执行性回答。
3. 采用 LLM 优先切片，但要有稳定可控的规则回退。

---

## 2. 1.2 的关键问题

1. 知识隔离问题：用户问 A 游戏，系统召回 B 游戏内容。
2. 切片质量问题：技巧步骤被切断，召回后无法直接执行。
3. 查询表达问题：用户口语化问题导致检索偏差。
4. 证据质量问题：召回片段重复或主题偏移。
5. 多轮稳定性问题：长会话中上下文漂移。

---

## 3. 交付范围（对应 6 个子模块）

1. gameKey 隔离与知识模型改造
2. LLM 优先切片与结构化校验
3. 意图识别与问题重写
4. 会话记忆摘要
5. 检索后处理与回答护栏
6. 可观测、评测与灰度上线

---

## 4. 模块依赖顺序

建议执行顺序：

1. 12：先做 gameKey 隔离（否则后续能力都没有正确过滤基础）
2. 13：再做 LLM 优先切片（提升知识质量）
3. 14：做意图重写与记忆摘要（提升 query 质量）
4. 15：做后处理与回答护栏（提升最终回答稳定性）
5. 16：做指标、压测、灰度上线（闭环验证）

---

## 5. 非目标（明确不做）

- 不做多模态输入（图像/视频战术分析）。
- 不做复杂权限租户体系。
- 不做自动在线学习回写。
- 不做分布式检索集群。

---

## 6. 统一技术约束

1. 所有新增功能必须可配置（可开关、可降级）。
2. 所有关键阶段必须发出 SSE 事件用于可观测。
3. 所有跨模块对象需保持统一命名（gameKey、intentType、riskFlags 等）。
4. 所有策略变更必须能通过配置回滚。

---

## 7. 建议里程碑与时间盒

- M1（2~3 天）：gameKey 隔离 + 上传入库改造
- M2（3~5 天）：LLM 优先切片 + 回退 + 校验
- M3（2~3 天）：意图识别 + 问题重写
- M4（2~3 天）：检索后处理
- M5（2 天）：会话记忆摘要
- M6（2~3 天）：联调评测 + 灰度

---

## 8. 完成定义（DoD）

1. 代码层：对应模块代码、配置、测试均完成。
2. 文档层：接口变化、配置说明、验收结果可追踪。
3. 运行层：端到端可演示，且关键指标满足阈值。

---

## 9. 阅读顺序

- 先读 12（隔离）
- 再读 13（切片）
- 再读 14（意图 + 记忆）
- 再读 15（后处理 + 护栏）
- 最后读 16（评测 + 上线）
